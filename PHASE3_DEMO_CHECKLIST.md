# CRPMS — Phase 3 Demo Checklist

This is the comprehensive pre-demo checklist and script for presenting the Continuous Risk-Aware Portfolio Management System (CRPMS) to faculty evaluators.

## SECTION 1 — 24 Hours Before Demo

*   **Code Freeze**: `git checkout develop && git pull origin develop`
*   **Dependencies**: `pip install -r requirements.txt` (ensure `openai` and `channels` are installed)
*   **Database Sync**: `python manage.py migrate` (must show no pending migrations)
*   **Integration Check**: `python manage.py run_integration_test`
    *   *Must show `12/12 PASS`.*
    *   *If any FAIL: list who is responsible (Data Ingestion, Feature Engine, Agents, etc.) and what to fix.*
*   **Base Data Load**: `python manage.py load_sector_data`
*   **Demo Data Seed**: `python manage.py seed_demo_data`
*   **Database Verification**: Open Django Admin and verify it shows 6 positions, 10 tickers, 2 alerts, and 1 backtest.
*   **Gemini API Test**: Run `python manage.py shell` and test `LLMClient.generate()` manually to confirm API key validity.
*   **Services Spun Up**: Start all 4 core services and leave them running overnight:
    1.  WSL Redis (`sudo service redis-server start`)
    2.  Django server (`python manage.py runserver`)
    3.  Celery worker (`celery -A config worker -l info`)
    4.  Celery beat (`celery -A config beat -l info`)

## SECTION 2 — 1 Hour Before Demo

*   **Services Check**: Verify all 4 services are still running (check terminal windows for errors/tracebacks).
*   **Frontend Check**: Open http://127.0.0.1:8000 — must load without errors.
*   **Admin Check**: Open http://127.0.0.1:8000/admin — verify seed data is visible.
*   **WebSocket Test**: Open the browser console on the dashboard and test the Channels connection:
    ```javascript
    const ws = new WebSocket('ws://localhost:8000/ws/portfolio/1/');
    ws.onmessage = (e) => console.log(e.data);
    ```
    *Must show:* `{"type": "connection", "status": "connected", "portfolio_id": 1}`
*   **Clean Slate**: Clear browser cache to ensure changes from last night reflect properly.
*   **Rehearsal**: Do one full dry run of the 10-minute demo script (below).

## SECTION 4 — Backup Plans

*   **If Gemini API is down**: Set `LLM_ENABLED=False` in `config/settings.py` and restart Django. The system will automatically activate the local template fallback. The demo still works perfectly.
*   **If Groq is also down**: Same as above — f-string templates take over. Explain the graceful fallback design as a deliberate resilience feature.
*   **If Redis is down**: Run `sudo service redis-server start` in WSL, then restart Celery.
*   **If WebSocket does not connect**: Demo the dashboard with manual page refreshing instead. Explain the underlying Channels broadcast architecture verbally.
*   **If Celery is not running**: Manually call the missing tasks from the Django shell (`python manage.py shell`) live during the demo.

## SECTION 5 — Faculty Questions + Answers

**1. Why PostgreSQL and not SQLite?**
SQLite locks the entire database during writes, which breaks down when Celery workers and WebSocket consumers are reading and writing simultaneously. PostgreSQL handles concurrent connections and JSONB fields, which we use heavily for agent outputs.

**2. Why Celery and not a cron job?**
Cron jobs cannot easily share state, retry failed API calls, or route tasks to different queues. Celery allows us to decouple heavy ML/LLM workloads from the HTTP request-response cycle.

**3. How does the Market Risk Agent classify stocks?**
It evaluates volatility (beta, standard deviation) and technical momentum indicators generated by the Feature Engine, mapping the quantitative score into one of four qualitative bands (LOW to CRITICAL).

**4. Why VADER first, then FinBERT?**
VADER is extremely fast and lightweight, allowing us to filter out noise from thousands of news headlines in bulk. FinBERT is computationally expensive, so we only invoke it on the pre-filtered, highly relevant financial headlines.

**5. What does the Drawdown Guard actually do, mechanically?**
It monitors the portfolio's total value against a 90-day peak high-water mark; when a 10% drop occurs, it intercepts the State Engine and forces all "INCREASE" or "REALLOCATE" agent decisions to "HOLD" until the drawdown recovers to 7%.

**6. How is capital allocation weighted — is it equal weight?**
No, the Capital Allocator uses a dynamic approach weighing both the fundamental Opportunity Score and the Risk Band inverted scalar, ensuring riskier assets receive proportionally less initial capital.

**7. Why Gemini API and not a locally hosted model?**
Local LLMs requiring the reasoning complexity needed for portfolio analysis (e.g., Llama 3 70B) demand server-grade GPUs (e.g., 2-4x A100s) which exceeds student hardware constraints. Gemini 2.0 Flash offers superior speed and reasoning via API.

**8. What is the difference between Mode 1 and Mode 2?**
Mode 1 monitors existing positions and recommends surgical actions (Hold/Reduce/Exit) based on live risk. Mode 2 handles fresh capital deployment, building a multi-position allocation plan from scratch across preferred sectors.

**9. How would this system work in real production?**
It would connect to a live broker API (like Zerodha Kite Connect) via a new Execution Engine to place the trades automatically once the user approves the Agent's recommendations on the dashboard.

**10. How is this different from just using a stock screener?**
A screener provides static, point-in-time filters based on hard limits. CRPMS is an active state machine that continuously monitors temporal risk, logs human-readable reasoning, and mathematically manages position sizing dynamically.

**11. What is the role of Redis in this system?**
Redis serves a dual purpose: it acts as the message broker for Celery tracking async tasks, and it powers the Django Channels channel layer for real-time WebSocket broadcasting to the dashboard.

**12. What happens if the Gemini API is unavailable during trading hours?**
The `SuggestionGenerator` catches the `LLMAPIError`, logs a warning, and automatically falls back to our local f-string templating engine. The user still receives actionable trade recommendations without interruption.
